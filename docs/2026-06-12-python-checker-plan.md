# Python Checker Implementation Plan

This is the dated implementation plan that task 8 of the
[Enforcement and Adoption Plan](2026-06-11-enforcement-and-adoption-plan.md)
promises. That entry fixes scope and contract; this document fixes the build:
the config schema, module layout, porting strategy, and sequencing. Where the
two disagree, the enforcement plan wins.

## Config Schema

The shared `slophammer.yml` gains a `python:` section. All four checkers
validate it strictly:

```yaml
python:
  coverage:
    threshold: 85        # >= 85, like every other ecosystem
    paths:
      - python/src
    exclude: []          # ExcludeEntry: string or {pattern, reason}
  complexity:
    max: 8
  dry:
    max_findings: 0
    paths:
      - python/src
    exclude: []
    copied_blocks:
      enabled: true
      min_tokens: 100
  mutation:
    targets:
      - python/src/slophammer_py/rules
    exclude: []
  dependency_boundaries:
    - from: python/src/slophammer_py/rules
      allow:
        - python/src/slophammer_py/config
  typecheck:
    demotions:           # reasoned ty rule demotions; bare demotion in
      - rule: deprecated # ty.toml/[tool.ty]/CLI flags is a finding
        reason: upstream false positive on sqlalchemy decorators
```

`migrations/` joins the conventional non-production list for Python paths.

## Module Layout

`python/` with src layout, `pyproject.toml`, uv-managed, committed `uv.lock`.
Python 3.12 minimum. PyYAML (`safe_load` only) is the single runtime
dependency. Internal boundaries mirror the TypeScript checker one-to-one so
the conformance behavior has an obvious counterpart to diff against:

| Module                         | Mirrors (TypeScript)            |
| ------------------------------ | ------------------------------- |
| `slophammer_py/cli.py`         | `src/cli/cli.ts`                |
| `slophammer_py/app.py`         | `src/app/app.ts`                |
| `slophammer_py/baseline.py`    | `src/app/baseline.ts`           |
| `slophammer_py/scan.py`        | `src/scan/scan.ts`              |
| `slophammer_py/config.py`      | `src/config/config.ts`          |
| `slophammer_py/repo.py`        | `src/repo/repo.ts`              |
| `slophammer_py/report.py`      | `src/report/report.ts`          |
| `slophammer_py/dry/`           | `src/dry/`                      |
| `slophammer_py/rules/`         | `src/rules/`                    |
| `slophammer_py/toolconfig.py`  | new: ty/mypy/pyright config     |

Tests live in `python/tests/`, pytest, named after the TypeScript test files
they port so coverage gaps are visible by filename.

## Porting Strategy

Binding evidence (`repo.py`) is half the risk. It is ported function-by
-function from `typescript/src/repo/repo.ts` — same function names
(snake_cased), same test cases ported from `typescript/tests/` — so behavior
differences are diffable line-by-line. The same applies to `baseline.py`
against `baseline.ts`. Everything else follows the spec directly.

The `py.*` evidence matchers are uv-native: `uv run <tool>`, `uvx <tool>`,
`uv run --directory <dir> <tool>`, plus bare-tool and `python -m` forms.
Package-script-style expansion does not exist in Python; Makefile, Taskfile,
justfile, and script reachability come from the shared binding-evidence
semantics unchanged.

## Rule Set

Fifteen `py.*` rules plus the four shared `repo.*` rules, per the enforcement
plan entry: project, typecheck (ty/mypy/pyright), types-strict (the full ty
contract: no unreasoned demotion of stable default-error rules,
`error-on-warning`, the four ignore-default promotions,
`respect-type-ignore-comments = false`; mypy `strict`/`disallow_untyped_defs`
with pydantic plugin when applicable; pyright strict), lint + format (Ruff,
tool-agnostic), annotation coverage via Ruff ANN selection, test, coverage
(>= 85), complexity (<= 8), dry + native copied-block port, mutation
declared, suppressions-justified (`# noqa`, `# type: ignore`,
`# ty: ignore` need reasons; bare `# type: ignore` without a code is a
finding), dependency-audit, dependency-boundaries (parsed with `ast`),
typed-marker for published packages, and scope-incomplete with the shared
scope-union semantics (targets + coverage paths + dry paths + mutation
targets).

`specs/ty-rules.json` is generated by `scripts/extract-ty-rules.py` from a ty
source checkout (`declare_lint!` blocks: rule, default level, stability) and
committed; refreshing it is part of bumping the supported ty version.

## Sequencing

1. Step 0 — the `python:` schema lands in the Go, TypeScript, and Rust
   validators and `specs/CONFIG.md`, so configs carrying the section stop
   failing strict validation. Independently correct; everything else
   depends on it.
2. Core machinery: scan → config → binding evidence → report/baseline → CLI,
   with ported tests green at each step.
3. Rules: `repo.*` first (conformance prerequisites), then `py.*`, then the
   ty contract and tool-config evidence.
4. Specs and fixtures: `rules.json`/`RULES.md`/`CONFIG.md` rows;
   `python-clean`, per-rule `python-missing-*`, `python-bad-dependency`,
   `python-demoted-rule`, `python-soft-warnings`; conformance registration
   via `uv run --directory python slophammer-py`.
5. Own gates: ty configured to the contract it enforces, Ruff with the ANN
   selection, pytest coverage >= 85, complexity <= 8, DRY 0, mutation
   declared (mutmut), `python-implementation` CI job, `make check-python`.
6. Release: `python-release.yml` on `python/v*` tags, PyPI trusted
   publishing, GitHub release from the workflow.

## Done Criteria

Identical to the enforcement plan entry: shared conformance passes with the
Python fixture set at parity, the three baseline exit-code cases pass against
the Python checker, and the checker passes its own gates in CI.
